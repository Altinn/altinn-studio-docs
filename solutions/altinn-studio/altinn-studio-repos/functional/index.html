Functionality in Repos

“Altinn Repositories” er en frittstende Git versjonskontroll og Application Lifecycle Management-lsning, som bde Designer, kodeeditorer og andre eksterne verkty vil benytte som back-end og felles sted for lagre og versjonere tjenestene som utvikles.

Repositories har et GitHub-lignende brukergrensesnitt og bde Git og REST-baserte tekniske grensesnitt.

Det vil vre en omfattende jobb forske utvikle en Git + ALM lsning p egen hnd, ref. all funksjonaliteten som behves.

Heldigvis finnes det flere open source “GitHub-kloner” som vi kan benytte, og blant dem virker Gitea vre mest lovende (uttales som /ti:/ - “gitty”). Bilder fra Gitea benyttes som eksempler videre i denne dokumentasjonen.

Kildekoden for Repositories ligger i VSTS
https://altinn.visualstudio.com/tjenester3.0/_git/AltinnCore

Funksjonalitet i “Repositories”

Repositories inneholder et repository med versjonerte filer for hver 3.0-tjeneste. Eksempel p typer filer som en “3.0-tjeneste” vil kunne best av er kode, tester, web-komponenter, layouts og stiler, oversettinger, arbeidsflyt og konfigurasjon.

Repositories vil tilby funksjonalitet rundt den enkelte 3.0-tjeneste, og eksponere funksjonaliteten via bde brukergrensesnitt og et REST-API som er en klone av GitHub sitt v3 API-format. Dette er positivt da det finnes andre verkty som ogs sttter dette API-formatet, og da det kan forenkle det kunne erstatte Gitea som back-end med andre liknende Git-systemer.

3.0 Tjeneste

Repository som representerer en “3.0-tjeneste”, og inneholder alle filer som tjenesten bestr av.
Hver utgave av en tjeneste vil leve i sin egen mappe, slik at ett repository kan inneholde n eller flere utgaver av den samme tjenesten.

Repository management

Hndtering av repositories er en kjerneaktivitet, da hver tjeneste som utvikles vil lagres i sitt eget repository.

  • Create/delete repository
  • Link to repository
  • Rename repository
  • Fork repository
    • Muliggjr at man kan ha et “master”-repository med felles tjenester som flere andre kan benytte, slik som dagens “master-tjenester” bare mye kraftigere
    • Muliggjr ogs at utviklere kan lage som egen kopi og teste ut nye muligheter og lsninger p sitt eget omrde uten pvirke originalen
  • List repositories
  • Download/upload files
  • Download repo
  • Create files and folders
  • Read/edit files
  • Rename files
  • Delete files
  • View raw file
  • Link to file

Dokumentering

Det er viktig ha god sttte for dokumentere tjenestene man utvikler, slik at det vil vre enkelt for andre personer p teamet bidra. Det er derfor nskelig ha innebygd Wiki-funksjonalitet med sttte for editere i markdown, slik som i Altinn docs og AltinnPedia.

  • Wiki
    • Add/edit/delete pages
    • Markdown editing
    • Link to specific wiki page
    • Support for other external wikis
    • Emojis
  • Markdown editing
    • Preview
    • Fullscreen
    • Side-by-side

Versjonering

Hver eneste fil som en 3.0-tjeneste bestr av skal ha full versjonshistorikk, uavhengig av om man jobber mot filene via Designer eller eksternt og distribuert/lokalt i en kodeeditor p egen maskin. Repositories sitt versjonskontrollsystem, som er basert p Git, er laget nettopp med tanke p dette.

  • Version history of all files, artifacts, code and configuration
    • Link to each change with a unique id
    • Link to each changed file
    • Link to who did the change
    • Highlight the last change done in the repository
    • Time of change
    • Comments/messages
  • Diff between versions

Release management

En svrt viktig egenskap nr man utvikler programvare (noe som 3.0-tjenester er) er kunne tagge ulike versjoner, og kunne g tilbake og teste eller patche en tidligere versjon hvis man nsker det. Det er ogs viktig kunne se alt som har endret seg mellom versjonene.

Siden Git benyttes for versjonering, s vil en release/tag som opprettes peke p en eksakt versjon av hver eneste fil som en 3.0-tjeneste bestr av, og man kan enkelt se hva som er endret.

  • Create tags for the 3.0 service
  • Link to tags
  • Create/delete releases
  • Link to releases
  • Marking of pre-releases
  • Time of release creation
  • User that triggered the release
  • Download zipped files of release
  • Browse content of a given tag/release

QA

En QA-prosess for godkjenning av endringer basert p branching og pull requests, vil for strre teams kunne vre mulig. Dette er i trd med hvordan programvare utvikles, inkludert selve Altinn.

Backlog management

Som for all annen programvareutvikling, s vil hver 3.0-tjeneste typisk ha en egen backlog med bugs, forbedringer, fremtidige features, etc. som tjenesteutviklerne vil jobbe mot. Dette er en sentral del av Application Lifecycle Management.

  • Create/close issues pr. 3.0 service
    • Time of creation
    • Who
  • Filter issues by open/closed
  • Add labels
    • Filter issues by label
  • Assign to milestone
    • Filter issues by milestone
  • Assign to user
    • Filter issues by assignee
  • Markdown editing
  • Discussions
  • Add files to issue
  • Notifications when issues change
  • Issues search
  • List issues

Org management

Organisasjoner er tjenesteeiere, men i 3.0 er det ikke noe i veien for at brukere ogs kan utvikle tjenester. I en organisasjon s kan man gi tilganger til brukere, og man kan opprette teams med tilgang til en eller flere 3.0-tjenester.
Det hndtere organisasjonen, som alt annet, skal vre 100% selvbetjent.

  • Create/delete organizations
    • Name/shortname
    • URL
    • Logo
  • List organizations
  • Link to organization
  • Create repository for organization ref. repository management
  • Add/remove teams in organization
  • List teams in organization
  • Add/remove users in organization
  • List all users in organization
  • Add/remove users in teams
  • List users in teams

Brukerstyring

Brukerne i denne sammenheng er tjenesteutvklerne. Det hndtere organisasjonen skal vre helt selvbetjent.

  • Add/remove users
  • Name
  • Avatar
  • Add/remove users to/from teams
  • “Starring” of favorite repositories
  • List of user repositories (including forks)
  • Activity log for user
  • Workspace for user (dette er veldig viktig, betyr at tjenesteutviklere kan “leke seg” p sitt eget omrde)

I frste omgang (MVP) vil tjenesteeier (org-admin) legge til alle sine brukere i gruppen “owner” slik at de fr tilgang til opprette repositories (tjenester). Gruppen “owner” gir vide fullmakter p organisasjonsniv (dvs tjenesteeierniv), s p sikt vil det implementeres mer granulert tilgangsstyring.

Ske

Tilgangsstyring

Det skal vre mulig styre tilgang til repositories, les/skriv/admin, teams, brukere, samt opprette private repos.

  • Access to 3.0 services (read/write)
  • Access to orgs
  • Users access
  • Private/public service-repos
  • Administrator
  • AM/AD-integration
  • API access

Trigge hendelser

I dagens hverdag, med bruk av Slack og andre systemer ifbm. utvikling, s er det viktig at “Repositories” kan trigge hendelser i andre systemer, f.eks. poste til en Slack-kanal nr en endring skjer i en 3.0-tjeneste.

  • Trigger events at org-level
  • Trigger events at 3.0 service-level

Logge inn

Tjenesteutviklere og andre interessenter skal kunne logge inn og f tilgang til organisasjoner og 3.0-tjenester.

  • Login page
  • CAPTCHA?
  • Forgot password?
  • 2-factor auth
  • External authentication
